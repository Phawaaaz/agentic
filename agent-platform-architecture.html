01Overview & goals

A control plane for AI agents: a single place for the team to define agents, run and observe sessions, and deploy agents as callable endpoints — independent of which LLM provider sits underneath.

Scope

Team-internal tool

Built for one engineering team, not external customers. No multi-tenant billing or public sign-up.

Providers

Multi-provider by design

Anthropic, OpenAI, Google, and self-hosted models all sit behind one internal interface.

Pillars

Agents, sessions, deployments

Three co-equal surfaces: build agents, observe runs, ship agents as endpoints.

Posture

Built to scale, not over-built

Stateless services and an upgrade path for storage — without day-one infrastructure most teams don't need yet.

02Core concepts

Four entities form the backbone of the system. Understanding how they relate is the key to everything that follows.

Agent

The definition or template. Holds a name, system prompt, model configuration (provider, model string, temperature, max tokens), and tool/MCP bindings. Agents themselves never change — they exist only as an identity that versions attach to.

Agent version

Every edit to an agent's configuration creates a new immutable version rather than overwriting the existing one. Versions are append-only: once a version has been used by any session or deployment, it is never modified. This is the single most important design decision in the system — it means a config change can never silently alter the behavior of something already running in production.

Session

A runtime instance of an agent version handling one conversation or task. Holds status (active, idle, completed, failed), full message/event history, token usage, and cost. This is the unit of observability — almost everything in the monitoring UI is a view over sessions.

Deployment

Binds a specific agent version to an environment (dev, staging, prod) and exposes it as a stable, callable endpoint. Because deployments pin to a version rather than to "the agent" in general, promoting a new version or rolling back is just repointing one foreign key.

Why versioning matters more than it seems to at first. Without it, editing an agent's prompt changes the behavior of every session and deployment using that agent immediately and silently. With it, you always know exactly which prompt and model configuration produced any given session, and rolling back a bad change is a one-line operation instead of an incident.

Requests enter through the gateway, which handles authentication and rate limiting before reaching the control plane. The control plane manages everything except the actual model calls — those are handed off to the execution layer, which runs asynchronously against a queue. This separation is the architectural decision that does the most work: it means heavy, slow agent runs never block the fast API surface the dashboard depends on.

04The control plane

The control plane is a set of stateless services — agent, session, deploy, and auth — that hold no in-memory state between requests. Everything persists to Postgres or the cache immediately. This is what makes scaling trivial: when traffic grows, you add replicas behind the gateway. No sticky sessions, no coordination between instances.

  • Agent service — CRUD for agent definitions and versions. Enforces the append-only versioning rule.
  • Session service — creates sessions, serves session history and live status to the dashboard.
  • Deploy service — promotes an agent version to an environment, manages the stable endpoint mapping.
  • Auth / RBAC — team membership and role scoping (who can edit vs. only view).
  • Secrets / vault — encrypted storage and retrieval of provider API keys.
  • Observability — ingests traces, metrics, and per-session cost data for the monitoring views.

05The execution layer

Agent runs — the loop of calling a model, executing a tool, calling the model again — happen in a separate worker pool fed by a queue, deliberately apart from the API servers.

This separation matters because agent runs are slow and bursty: a single run can take minutes and involve many model calls, while API requests need to return in milliseconds. Running both in the same process means one heavy agent run starves the dashboard. Splitting them lets the worker pool scale against queue depth, independent of API traffic.

The queue also provides durability for free. If a worker crashes mid-run, the task returns to the queue and another worker picks it up. For a multi-provider system this matters doubly, since provider rate limits and outages are routine — the queue absorbs them instead of surfacing them as user-facing failures.

06The provider adapter

Everything above this layer speaks one internal interface. Only the adapter knows that Anthropic streams differently from OpenAI, that tool-call formats differ between providers, or that token counting works differently per model family.

interface LLMAdapter {
  stream(messages, config) -> AsyncIterator<Chunk>
  countTokens(messages) -> number
  toolFormat(tools) -> ProviderTools
}

Concentrating provider-specific knowledge in one place means adding a new provider is a contained change to one layer, not a refactor spread across the codebase.

Build vs. buy. Writing every adapter by hand gives maximum control but costs real time. A normalizing gateway — LiteLLM (proxy giving every provider an OpenAI-compatible API) or the Vercel AI SDK (TypeScript, unified streaming and tool calling) — gets the same abstraction with fallbacks, retries, and spend tracking built in. For a team-internal tool, starting on one of these saves weeks; hand-written adapters can replace it later if you outgrow it.

07The data layer

The data layer splits by access pattern, which is the detail most teams get wrong until volume forces the issue.

Entity data — agents, sessions, deployments — is low-volume, relational, and needs transactions: that's Postgres. Session traces — every message, token count, latency figure, tool call — are high-volume, append-only, and queried analytically ("cost per agent this week"). Forcing both into the same table works until trace volume swamps the database and dashboard queries slow down.

The fix is to keep traces in their own table from day one (session_events, see schema below) so the write pattern is already isolated, then migrate that table to a columnar store like ClickHouse — or partition it within Postgres — once volume actually demands it. You don't need ClickHouse on day one; you need the table boundary to already exist so the migration is a data move, not a redesign.

Part three — data model

08Data model & schema

Seven tables: four core entities, plus credentials, trace events, and team ownership.

TablePurposeKey columns
teamsTop-level ownership boundaryid, name, created_at
agentsIdentity only — name and slug, never editedid, team_id, name, slug
agent_versionsImmutable config snapshot per editid, agent_id, version_number, system_prompt, provider, model, tool_config, model_params
deploymentsBinds a version to an environment + endpointid, agent_version_id, environment, endpoint_slug, status
sessionsOne runtime conversation/task instanceid, agent_version_id, deployment_id, status, total_tokens, cost_usd
session_eventsAppend-only trace of every message/tool callid, session_id, event_type, payload, tokens, latency_ms
provider_credentialsEncrypted provider API keys, scoped to teamid, team_id, provider, encrypted_key

Why agents and agent_versions are split

agents is an identity row that never changes. All real configuration lives in agent_versions, where every edit inserts a new row. A deployment's agent_version_id always points at something immutable — promoting a version is changing one foreign key, and rolling back is pointing it at the previous one.

Why sessions carries both agent_version_id and deployment_id

A session can originate from two paths: someone testing an agent directly in the builder (no deployment), or a deployed endpoint being invoked (has a deployment). Making deployment_id nullable lets one table serve both paths, so the session-monitoring view never needs a union query across two tables.

Why session_events is its own table, not a JSON column on sessions

It's tempting to add a messages jsonb column on sessions and append to it. Avoid this. Postgres rewrites the entire JSON blob on every append, which slows down as conversations grow, and a blob can't be queried at the event level — exactly the query you'll want once there's real usage ("show every failed tool call this week"). A separate table with an event_type column (message, tool_call, tool_result, error) keeps writes small and queryable, and is also the table to move into ClickHouse first when volume grows.

Why provider_credentials stores encrypted_key, never a raw key

Provider API keys should never sit in plaintext, even in a database only the team can reach. Use envelope encryption: a master key held in a KMS (AWS KMS, GCP KMS, or HashiCorp Vault) encrypts each provider key before it's written, and the app decrypts on read via a KMS call rather than reading a stored key directly. A database leak alone then doesn't leak provider credentials.

Add later, not now: a dedicated tool/MCP-binding table once agents bind to many external tools (a JSON column on agent_versions is fine until you need to query "which agents use this tool"); a team_members join table with roles once more than one permission level is needed; an index on (deployment_id, started_at) on sessions as soon as the monitoring view filters by deployment and time range — which will be almost immediately.

09Scaling notes

In rough order of when each becomes a real problem:

  1. Trace volume overwhelming the primary database — mitigated by keeping session_events as its own table from the start, migrating to ClickHouse or partitioned Postgres when volume demands it.
  2. Long-running agent runs holding worker capacity — mitigated by scaling workers against queue depth and setting per-run timeouts.
  3. Provider rate limits cascading into failures — mitigated by the queue plus per-provider concurrency caps inside the adapter layer.
  4. Cost visibility lagging reality — mitigated by metering tokens at the adapter and streaming usage to the observability store in real time rather than in a nightly batch.

The most common overengineering trap at this stage is reaching for Kubernetes or a full ClickHouse + Grafana stack on day one. Both have clear upgrade paths from the simpler choices above — start simple, migrate the specific piece that hits its ceiling, not the whole stack at once.

11Monorepo structure

A monorepo fits because the dashboard, API, and worker share the same TypeScript types (Agent, Session, Deployment), and a schema change should update all three without juggling package versions across separate repos.

Tooling: Turborepo + pnpm workspaces. Turborepo caches builds and runs tasks in parallel across packages without the configuration overhead of Nx — appropriate weight for a repo that won't have 15+ packages. Skip Lerna; it's legacy.

agent-platform/
├── apps/
│   ├── dashboard/      # Next.js — agent builder, session monitor, deploy UI
│   ├── api/            # Fastify/Hono — control plane (agents, sessions, deployments, auth)
│   └── worker/         # BullMQ worker — agent execution loop
├── packages/
│   ├── db/             # Drizzle schema + migrations + typed client
│   ├── adapter/        # Provider adapter — unified LLM interface
│   ├── types/          # Shared TS types: Agent, Session, Deployment, etc.
│   └── ui/             # Shared shadcn/ui components
├── turbo.json
├── pnpm-workspace.yaml
└── package.json

packages/db is the package every app touches — the dashboard reads sessions for monitoring, the API writes agents and deployments, the worker writes session_events. Defining the schema once means a column rename surfaces as a type error everywhere it's used, instead of a runtime bug discovered later.

packages/adapter stays separate from apps/worker so provider behavior can be unit-tested (mocking Anthropic vs. OpenAI responses) without spinning up the queue, and so the API can call it directly too — for example, a "test this agent" button that shouldn't need the full async worker path.

pnpm dlx create-turbo@latest agent-platform
cd agent-platform
pnpm add -D typescript -w

12Build sequencing

Build the provider adapter and the Agent/Session models first, and get one agent running end to end with live session traces visible. That alone is already useful to the team. Layer the builder UI next, then deployment last.

Trying to build all three pillars fully in parallel tends to stall progress on all of them. The data model has to account for all three from the start — hence pinned agent versions in the schema above — but the build order doesn't have to match the data model's symmetry.
