01Overview & goals

A control plane for AI agents: a single place for the team to define agents, give them tools, skills, and memory, run and observe sessions, and deploy agents as callable endpoints — independent of which LLM provider sits underneath.

Scope

Team-internal tool

Built for one engineering team, not external customers. No multi-tenant billing or public sign-up.

Providers

Multi-provider by design

Anthropic, OpenAI, Google, and self-hosted models all sit behind one internal interface.

Pillars

Agents, sessions, deployments

Three co-equal surfaces: build agents, observe runs, ship agents as endpoints.

Capabilities

Tools, skills & memory

Agents act through tools, load reusable skills, and carry memory across sessions — consolidated by a background dreaming process.

One thing to settle before building any of this: does a given task even need an agent? There's a ladder — a single model call for well-specified tasks (classify, extract, summarize), a code-orchestrated workflow for multi-step pipelines whose control flow you can write, and a model-driven agent only when the trajectory can't be specified in advance. Reach for the full agent setup below only when the work genuinely can't be scripted ahead of time; everything else is cheaper and more reliable a rung down.

02Core concepts

Six entities form the backbone of the system. Understanding how they relate is the key to everything that follows.

Agent

The definition or template. Holds a name, system prompt, model configuration (provider, model string, effort/thinking, max tokens), and bindings to tools, skills, and memory scope. Agents themselves never change — they exist only as an identity that versions attach to.

Agent version

Every edit to an agent's configuration creates a new immutable version rather than overwriting the existing one. Versions are append-only: once a version has been used by any session or deployment, it is never modified. This is the single most important design decision in the system — it means a config change can never silently alter the behavior of something already running in production.

Session

A runtime instance of an agent version handling one conversation or task. A session is an event stream, not just stored text: it moves through a lifecycle — running ↔ idle → completed/failed — and emits typed events as it goes (model messages, thinking, tool calls and their results, status transitions). When it pauses it carries a stop reason (end_turn for done, tool_use/requires_action when it's waiting on a tool result or a human approval, refusal, max_tokens) that tells the caller what to do next. This is the unit of observability, and the raw material memory is later distilled from. The persisted form of that stream is the session_events table (§12).

Environment

A reusable template for the sandbox an agent's tools run in: the networking policy (open egress vs. a host allowlist), preinstalled packages, and resource limits. Environments are kept separate from agents on purpose — one environment ("data analysis, no network") can back many agents, and you can change the sandbox without touching agent config. A session names both an agent version and an environment.

Container

The actual isolated compute instance a session gets, provisioned from its environment. This is where tools execute — bash, file edits, code — not where the model loop runs. Keeping the model loop and the tool sandbox apart is the core safety boundary: the model decides, the container acts, and a misbehaving tool can't reach beyond its own container.

Deployment

Binds a specific agent version to an environment (dev, staging, prod) and exposes it as a stable, callable endpoint. Because deployments pin to a version rather than to "the agent" in general, promoting a new version or rolling back is just repointing one foreign key.

Why versioning matters more than it seems to at first. Without it, editing an agent's prompt, tools, or skills changes the behavior of every session and deployment using that agent immediately and silently. With it, you always know exactly which prompt, model configuration, tools, and skills produced any given session, and rolling back a bad change is a one-line operation instead of an incident.

03Agent capabilities

An agent is more than a prompt and a model. Four capabilities turn a model call into an agent that acts, specializes, and improves over time: the tools it can act with, the skills it can load, the memory it carries, and the dreaming that forms that memory.

Tools — what an agent can do

A tool is a typed contract — a name, an input schema, and a handler — that lets the model act on the world. Tools come in three shapes: functions you host (your code answers the call), server tools the model runs directly (code execution, web search), and MCP servers — a standard protocol (the Model Context Protocol) for exposing third-party capabilities like GitHub or a database. Tools are registered once and bound to agent versions, so one search_docs tool can back many agents. Whether a given tool runs automatically or pauses for human approval is a per-tool permission policy — see §08.

Skills — what an agent knows how to do

A skill is a packaged, reusable capability: a bundle of instructions — and optionally the tools and resources that go with them — that teaches an agent a specific kind of task ("triage a support ticket", "generate an invoice"). Concretely it's a folder with a manifest (a SKILL.md) whose short description stays in context by default; the agent loads the full instructions only when the task calls for it. That progressive disclosure is the point — an agent can have many skills available without paying for all of them in every prompt. Skills are versioned and append-only like agents, and shareable across agents, so a capability is authored once instead of copy-pasted into every system prompt.

Memory — what an agent remembers

Sessions are ephemeral; memory is what survives them. Two kinds: episodic — distilled records of what happened in past sessions (decisions, outcomes) — and semantic — durable facts the agent has learned (a user's preferences, a project's constraints). Memory is scoped (per agent, per user, or per team) and recalled by relevance at the start of and during a session, so an agent walks in already knowing what it learned last time. This is distinct from managing the context window within one session — see §10.

Dreaming — how memory is formed

Raw session history is not memory. Saving every event is hoarding; saving nothing means the agent never learns. Dreaming is the offline process in between: while an agent is idle, a background job replays recent sessions and reflects — extract what's worth keeping, merge what duplicates an existing memory, prune what's gone stale. The name is deliberate. Like sleep consolidation, it's the step where experience becomes knowledge, and it's what keeps the memory store curated rather than either empty or unbounded.

Why capabilities bind to a version, not to the agent. Tools and skills attach to an agent version, the same immutable snapshot that pins the prompt and model config. Changing an agent's toolset or skills creates a new version rather than mutating the old one — so you can always reconstruct exactly which tools and skills produced any given session, and roll a bad change back the same way you roll back a prompt.

Requests enter through the gateway, which handles authentication and rate limiting before reaching the control plane. The control plane manages everything except the actual model calls and tool execution — those are handed off to the execution layer, which runs asynchronously against a queue, with each session's tools executing in its own container. Memory recall happens synchronously in the control plane; memory formation happens off to the side in the scheduled dreaming worker. This separation is the architectural decision that does the most work: heavy, slow agent runs — and slow consolidation passes — never block the fast API surface the dashboard depends on.

05The control plane

The control plane is a set of stateless services — agent, session, deploy, auth, the tool/skills/memory registries, and the vault — that hold no in-memory state between requests. Everything persists to Postgres or the cache immediately. This is what makes scaling trivial: when traffic grows, you add replicas behind the gateway. No sticky sessions, no coordination between instances.

  • Agent service — CRUD for agent definitions and versions. Enforces the append-only versioning rule.
  • Session service — creates sessions, tracks their lifecycle, and serves the event stream and live status to the dashboard.
  • Deploy service — promotes an agent version to an environment, manages the stable endpoint mapping.
  • Tool registry — the catalog of available tools and their input schemas; supplies the worker with the set of callable tools for an agent version.
  • Skills registry — stores skill definitions and versions, resolves which skill versions an agent version loads.
  • Memory service — recall on the read path: retrieves the most relevant memories for a scope and serves them to the worker at session start.
  • Vault — stores tool and provider credentials and injects them at egress, so secrets never enter a container (§08).
  • Auth / RBAC — team membership and role scoping (who can edit vs. only view).
  • Observability — ingests traces, metrics, and per-session cost data for the monitoring views.

06The agent loop & loop engineering

An agent run is a loop, not a single call: gather context → take an action → observe the result → repeat, until the model decides it's done. A worker drives that loop — it calls the model, and when the model asks for a tool, the worker runs it (in the session's container) and feeds the result back as the next turn. The loop ends when the model returns a final answer instead of a tool call.

A session exposes that loop as an event stream. Rather than returning one blob at the end, the worker emits typed events as they happen — model text, thinking, each tool call and its result, and status transitions — which the dashboard streams live and the trace store persists. This is why session monitoring is a view over session_events rather than a single response column.

The session's status and stop reason are what callers branch on:

  • running — the worker is actively driving the loop.
  • idle + end_turn — the agent finished and is awaiting the next message.
  • idle + requires_action — the loop is blocked on something only the caller can provide: a custom tool result, or a human approval for a gated tool. The caller responds, and the loop resumes.
  • failed / refusal — terminal; surface it rather than retrying blindly.
Idle is not the same as done. A session goes idle every time it needs something from the outside — a tool confirmation, a custom-tool result, the next user message. Treating every idle as "finished" is the most common way to build a broken agent client: check the stop reason, and only stop when it's terminal (end_turn, failed) rather than a requires_action pause that's waiting on you.

Loop engineering

The loop isn't fixed plumbing — it's the part of an agent you tune the most, and the thing that most decides whether an agent is reliable or flaky. Loop engineering is the discipline of designing each pass: what the agent sees on a turn, how it checks its own work, and when it's allowed to stop. The levers that matter:

  • Context per turn — feed the agent what's relevant now, not the entire history. Too little and it forgets the goal; too much and it gets slow, expensive, and distracted. Recall, compaction, and context editing (§10) all feed this decision.
  • Verification — give the agent a way to check its own output before it calls the work done: run the tests, re-read the file it just edited, compare the result against the goal. An agent that never verifies confidently ships wrong answers.
  • Stopping conditions — define what "done" looks like and cap the loop, so a confused agent doesn't spin forever burning tokens. A per-run timeout and a max-turns limit are the cheap backstops.
  • Error recovery — when a tool fails, hand the error back to the model as the next turn instead of crashing the run. The model usually adjusts and retries; the queue (§07) absorbs the failures it can't.

None of this changes the data model — it's behavior that lives in the worker. But it's where most of the effort goes once the platform itself works, which is exactly why the worker is a package you can test in isolation rather than logic buried in the API.

07The execution layer

The agent loop (§06) runs in a separate worker pool fed by a queue, deliberately apart from the API servers.

This separation matters because agent runs are slow and bursty: a single run can take minutes and involve many model calls, while API requests need to return in milliseconds. Running both in the same process means one heavy agent run starves the dashboard. Splitting them lets the worker pool scale against queue depth, independent of API traffic.

The queue also provides durability for free. If a worker crashes mid-run, the task returns to the queue and another worker picks it up. For a multi-provider system this matters doubly, since provider rate limits and outages are routine — the queue absorbs them instead of surfacing them as user-facing failures. The same worker infrastructure runs the scheduled dreaming job (§10), so memory consolidation reuses the queue's retries and concurrency limits rather than introducing a second scheduler.

08Tools, processing & permissions

The agent worker decides to call a tool; the processing layer is where that call actually runs. Keep it separate from the provider adapter (§09): the adapter talks to models, the processing layer talks to tools.

  • Tool registry — the catalog of tools and their input schemas. The worker reads it to tell the model what's callable, and validates every tool call against the schema before running it.
  • MCP gateway — routes MCP tool calls to the right server. An agent declares the MCP servers it uses (name + URL, no credentials); the gateway is the single egress point, and the vault supplies the auth per session.
  • Container / sandbox — where handlers execute. Code-running tools run in the session's container with no ambient credentials, per-call timeouts, and concurrency caps.

Permission policies — the human-in-the-loop gate

Every server-executed tool carries a policy: always_allow (the call runs automatically) or always_ask (the session goes idle with stop reason requires_action and waits for an explicit confirmation before the call runs). This is how you put a human in front of irreversible actions — a send_email or delete_record tool asks; a read_file doesn't. The model emits the call, the platform blocks, and the caller allows or denies — a denial can carry a reason that's handed back to the model so it can adjust.

Vaults — credentials the sandbox never sees

Tool and MCP credentials do not go into the container. They live in the vault, and a proxy injects them into the outbound request after it leaves the sandbox. So code running in the container — including anything the model itself wrote — can't read or exfiltrate a key, even under prompt injection. OAuth tokens auto-refresh; per-tool secrets are substituted at egress against an allowlist of hosts. This is a stronger guarantee than encrypting keys at rest (§12), and it's the right default for anything an agent can call.

Tools are the blast radius. A model call only produces text; a tool call changes state — it writes a row, sends a request, or runs code. Treat every handler as untrusted by default: run code in the sandbox, gate irreversible actions behind always_ask, keep credentials in the vault rather than handing the agent raw keys, set timeouts, and log every call as a tool_call/tool_result event so the session trace shows exactly what the agent did, not just what it said.

09The provider adapter

Everything above this layer speaks one internal interface. Only the adapter knows that Anthropic streams differently from OpenAI, that tool-call formats differ between providers, or that token counting works differently per model family.

interface LLMAdapter {
  stream(messages, config) -> AsyncIterator<Chunk>
  countTokens(messages) -> number
  toolFormat(tools) -> ProviderTools
  embed(texts) -> Vector[]          // memory recall reuses this
}

Concentrating provider-specific knowledge in one place means adding a new provider is a contained change to one layer, not a refactor spread across the codebase. The same adapter also produces the embeddings the memory subsystem recalls against, so there's one place that talks to providers — for completions and for embeddings alike.

Build vs. buy. Writing every adapter by hand gives maximum control but costs real time. A normalizing gateway — LiteLLM (proxy giving every provider an OpenAI-compatible API) or the Vercel AI SDK (TypeScript, unified streaming and tool calling) — gets the same abstraction with fallbacks, retries, and spend tracking built in. For a team-internal tool, starting on one of these saves weeks; hand-written adapters can replace it later if you outgrow it.

10Context & memory

Two different problems wear the word "memory." Within a single long session, the context window fills up. Across sessions, the agent needs to carry forward what it learned. They have different mechanisms — don't conflate them.

In-session: compaction & context editing

As a session's event history approaches the model's context-window limit, compaction summarizes the earlier turns into a compact block so the run can keep going — the summary replaces the raw history but preserves the thread. Context editing is the lighter-weight sibling: it prunes stale content (old tool results, completed thinking blocks) rather than summarizing, keeping the transcript lean turn over turn. Both operate inside one session, and neither persists anything beyond it. They're about keeping a single run affordable and coherent — not about learning.

Across sessions: recall, the memory store & dreaming

Durable memory is a store the agent reads and writes between sessions. In some stacks this is a file-backed memory tool — the agent reads and writes a /memories directory that survives restarts; here it's the recall + dreaming subsystem. The read and write paths are deliberately asymmetric:

Recall (read path). At session start, and optionally mid-session, the memory service retrieves the most relevant memories for the active scope (agent, user, or team) and injects them into context. Relevance is vector similarity over the memory store, weighted by recency and salience. This is a control-plane read — fast and synchronous, in the hot path of starting a session — so an agent begins already knowing what it learned before.

Formation (write path) — dreaming. Memory is not written synchronously during a session. Candidate observations accumulate, and a scheduled dreaming job — off the hot path — processes them per scope: distil episodic summaries, extract semantic facts, dedupe against existing memories by similarity, prune stale or superseded ones, and recompute salience. Running it offline keeps the session loop fast, lets consolidation use a cheaper model, and lets it look across many sessions at once — the cross-session view a synchronous write can never have.

Storage

Memories live in Postgres with an embedding column (pgvector) for similarity recall. Promote to a dedicated vector store only when recall latency or index size demands it — the same "keep the boundary, migrate later" move as session_events → ClickHouse in the data layer.

The two layers are complementary, not competing. Compaction and context editing keep one run within budget; the memory store makes the next run smarter. And dreaming is scheduled rather than per-session for a reason: consolidating ten sessions at once is cheaper and better than ten separate writes — it can see that the same fact showed up five times (raise its salience) or that today's session contradicts last week's (supersede the old memory). Schedule it on idle, bound each run, and record it as a dream_run so every change to memory is auditable.

11The data layer

The data layer splits by access pattern, which is the detail most teams get wrong until volume forces the issue.

Entity data — agents, sessions, deployments, tools, skills, environments — is low-volume, relational, and needs transactions: that's Postgres. Session traces — every message, token count, latency figure, tool call — are high-volume, append-only, and queried analytically ("cost per agent this week"). Forcing both into the same table works until trace volume swamps the database and dashboard queries slow down.

The fix is to keep traces in their own table from day one (session_events, see schema below) so the write pattern is already isolated, then migrate that table to a columnar store like ClickHouse — or partition it within Postgres — once volume actually demands it. You don't need ClickHouse on day one; you need the table boundary to already exist so the migration is a data move, not a redesign. Memory embeddings get the same treatment: an embedding column on memories in Postgres now, a dedicated vector store later, with the boundary already drawn.

Why agents and agent_versions are split

agents is an identity row that never changes. All real configuration lives in agent_versions, where every edit inserts a new row. A deployment's agent_version_id always points at something immutable — promoting a version is changing one foreign key, and rolling back is pointing it at the previous one.

Why environments are their own table, not fields on the agent

The sandbox config (networking, packages, limits) lives in environments rather than on agent_versions because it's a different axis of change with a different owner. One environment backs many agents; you tighten a network policy once instead of re-versioning every agent that uses it. A session records both agent_version_id and environment_id, so you always know which sandbox a run executed in.

Why tools and skills bind through join tables, with a permission policy

Tools and skills attach to an agent_version through join tables rather than JSON lists. The join tables make the reverse query — "which agents use this tool?" — a simple index lookup, exactly the question you ask before changing or deprecating a shared tool. agent_version_tools also carries the permission_policy (always_allow / always_ask), so the human-in-the-loop gate for a tool is pinned to the version, not set globally.

Why session_events is its own table, not a JSON column on sessions

It's tempting to add a messages jsonb column on sessions and append to it. Avoid this. Postgres rewrites the entire JSON blob on every append, which slows down as conversations grow, and a blob can't be queried at the event level — exactly the query you'll want once there's real usage ("show every failed tool call this week"). A separate table with an event_type column (message, thinking, tool_call, tool_result, status, error) keeps writes small and queryable, and is also the table to move into ClickHouse first when volume grows.

Why memories carries a scope and an embedding, and dream_runs exists

memories.scope_type + scope_id let the same table serve per-agent, per-user, and per-team memory without three near-identical tables — recall filters by scope, then ranks by vector similarity over embedding and by salience/last_used_at. dream_runs is the audit trail for formation: because dreaming mutates memory outside any session, you want a record of every pass — what it processed and what it changed — so a memory that suddenly disappears or shifts is traceable to the run that did it.

Why credentials are a vault, not a plaintext column

Provider and tool secrets are never stored in plaintext. vault_credentials holds them encrypted (envelope encryption via a KMS — AWS KMS, GCP KMS, or HashiCorp Vault), and the running system injects them into outbound requests at egress rather than handing them to the container (§08). A database leak alone then doesn't leak credentials, and the sandbox never holds a key it could exfiltrate.

Add later, not now: an index on (deployment_id, started_at) on sessions as soon as the monitoring view filters by deployment and time range — which will be almost immediately; a team_members join table with roles once more than one permission level is needed; a dedicated vector store for memories only once pgvector recall becomes the bottleneck; per-tool usage counters once you need to rank tools by adoption.

13Scaling notes

In rough order of when each becomes a real problem:

  1. Trace volume overwhelming the primary database — mitigated by keeping session_events as its own table from the start, migrating to ClickHouse or partitioned Postgres when volume demands it.
  2. Long-running agent runs holding worker capacity — mitigated by scaling workers against queue depth and setting per-run timeouts.
  3. Context windows filling on long sessions — mitigated by compaction and context editing inside the run, before the model hits its limit (§10).
  4. Provider rate limits cascading into failures — mitigated by the queue plus per-provider concurrency caps inside the adapter layer.
  5. Memory recall slowing the session hot path — mitigated by an embedding index in pgvector, scope-filtering before ranking, and graduating to a dedicated vector store only when the index outgrows Postgres.
  6. Dreaming cost growing with session volume — mitigated by running it on idle against a bounded batch, using a cheaper model, and processing per scope rather than over the whole corpus each pass.
  7. Cost visibility lagging reality — mitigated by metering tokens at the adapter and streaming usage to the observability store in real time rather than in a nightly batch.

15Monorepo structure

A monorepo fits because the dashboard, API, and worker share the same TypeScript types (Agent, Session, Deployment, Tool, Skill, Memory), and a schema change should update all three without juggling package versions across separate repos.

Tooling: Turborepo + pnpm workspaces. Turborepo caches builds and runs tasks in parallel across packages without the configuration overhead of Nx — appropriate weight for a repo that won't have 15+ packages. Skip Lerna; it's legacy.

agent-platform/

│   ├── dashboard/      # Next.js — agent builder, session monitor, deploy UI
│   ├── api/            # Fastify/Hono — control plane (agents, sessions, tools, skills, memory recall)
│   └── worker/         # BullMQ — agent loop + scheduled dreaming job
├── packages/

packages/db is the package every app touches — the dashboard reads sessions for monitoring, the API writes agents and deployments, the worker writes session_events and memories. Defining the schema once means a column rename surfaces as a type error everywhere it's used, instead of a runtime bug discovered later.

packages/adapter, packages/tools, and packages/memory stay separate from apps/worker so each can be unit-tested in isolation — mocking Anthropic vs. OpenAI responses, a tool handler, or a consolidation pass without spinning up the queue — and so the API can call them directly too: a "test this agent" button that shouldn't need the full async worker path, or a memory-recall call on the session read path.

16Build sequencing

Build the provider adapter and the Agent/Session models first, and get one agent running end to end — through the full loop, with live session events visible in the dashboard. Tools come in this same first push: an agent with no tools is just a chat box, so the tool registry, the sandbox, and the permission gate are part of "one agent running end to end," not a later add-on.

Memory and dreaming come next, once the loop works and there are real sessions to consolidate — you can't dream without session history. Recall is the half to ship first (it's read-only and immediately useful); the scheduled dreaming job follows once there's enough history to chew on. In-session context management (compaction, context editing) lands alongside the loop as soon as runs get long enough to approach the context limit.

Skills layer in once more than one agent needs the same capability — building the skills subsystem before that is premature abstraction. The builder UI and deployment come last.

Trying to build all of this fully in parallel tends to stall progress on all of it. The data model has to account for every capability from the start — hence pinned agent versions, environments, join tables for tools and skills, and the memory tables in the schema above — but the build order doesn't have to match the data model's symmetry.
