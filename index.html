01Overview & goals

A control plane for AI agents: a single place for the team to define agents, give them tools, skills, and memory, run and observe sessions, and deploy agents as callable endpoints — independent of which LLM provider sits underneath.

Scope

Team-internal tool

Built for one engineering team, not external customers. No multi-tenant billing or public sign-up.

Providers

Multi-provider by design

Anthropic, OpenAI, Google, and self-hosted models all sit behind one internal interface.

Pillars

Agents, sessions, deployments

Three co-equal surfaces: build agents, observe runs, ship agents as endpoints.

Capabilities

Tools, skills & memory

Agents act through tools, load reusable skills, and carry memory across sessions — consolidated by a background dreaming process.

02Core concepts

Four entities form the backbone of the system. Understanding how they relate is the key to everything that follows.

Agent

The definition or template. Holds a name, system prompt, model configuration (provider, model string, temperature, max tokens), and bindings to tools, skills, and memory scope. Agents themselves never change — they exist only as an identity that versions attach to.

Agent version

Every edit to an agent's configuration creates a new immutable version rather than overwriting the existing one. Versions are append-only: once a version has been used by any session or deployment, it is never modified. This is the single most important design decision in the system — it means a config change can never silently alter the behavior of something already running in production.

Session

A runtime instance of an agent version handling one conversation or task. Holds status (active, idle, completed, failed), full message/event history, token usage, and cost. This is the unit of observability — almost everything in the monitoring UI is a view over sessions. It is also the raw material memory is later distilled from.

Deployment

Binds a specific agent version to an environment (dev, staging, prod) and exposes it as a stable, callable endpoint. Because deployments pin to a version rather than to "the agent" in general, promoting a new version or rolling back is just repointing one foreign key.

Why versioning matters more than it seems to at first. Without it, editing an agent's prompt, tools, or skills changes the behavior of every session and deployment using that agent immediately and silently. With it, you always know exactly which prompt, model configuration, tools, and skills produced any given session, and rolling back a bad change is a one-line operation instead of an incident.

03Agent capabilities

An agent is more than a prompt and a model. Four capabilities turn a model call into an agent that acts, specializes, and improves over time: the tools it can act with, the skills it can load, the memory it carries, and the dreaming that forms that memory.

Tools — what an agent can do

A tool is a typed contract: a name, an input schema, and a handler that lets the model act on the world — query a database, call an API, run code, hit an MCP server. Tools are registered once and bound to agent versions, so the same search_docs tool can back many agents without being redefined for each.

Skills — what an agent knows how to do

A skill is a packaged, reusable capability: a bundle of instructions — and optionally the tools and resources that go with them — that teaches an agent a specific kind of task ("triage a support ticket", "generate an invoice", "run a deep-research pass"). Skills are versioned and append-only exactly like agents, composable (one agent loads many), and shareable across agents, so a capability is authored once instead of copy-pasted into every system prompt.

Memory — what an agent remembers

Sessions are ephemeral; memory is what survives them. Two kinds: episodic — distilled records of what happened in past sessions (decisions, outcomes) — and semantic — durable facts the agent has learned (a user's preferences, a project's constraints). Memory is scoped (per agent, per user, or per team) and recalled by relevance at the start of and during a session, so an agent walks in already knowing what it learned last time.

Dreaming — how memory is formed

Raw session history is not memory. Saving every event is hoarding; saving nothing means the agent never learns. Dreaming is the offline process in between: while an agent is idle, a background job replays recent sessions and reflects — extract what's worth keeping, merge what duplicates an existing memory, prune what's gone stale. The name is deliberate. Like sleep consolidation, it's the step where experience becomes knowledge, and it's what keeps the memory store curated rather than either empty or unbounded.

Why capabilities bind to a version, not to the agent. Tools and skills attach to an agent version, the same immutable snapshot that pins the prompt and model config. Changing an agent's toolset or skills creates a new version rather than mutating the old one — so you can always reconstruct exactly which tools and skills produced any given session, and roll a bad change back the same way you roll back a prompt.

Requests enter through the gateway, which handles authentication and rate limiting before reaching the control plane. The control plane manages everything except the actual model calls and tool execution — those are handed off to the execution layer, which runs asynchronously against a queue. Memory recall happens synchronously in the control plane on the read path; memory formation happens off to the side in the scheduled dreaming worker. This separation is the architectural decision that does the most work: it means heavy, slow agent runs — and slow consolidation passes — never block the fast API surface the dashboard depends on.

05The control plane

The control plane is a set of stateless services — agent, session, deploy, auth, and the tool/skills/memory registries — that hold no in-memory state between requests. Everything persists to Postgres or the cache immediately. This is what makes scaling trivial: when traffic grows, you add replicas behind the gateway. No sticky sessions, no coordination between instances.

  • Agent service — CRUD for agent definitions and versions. Enforces the append-only versioning rule.
  • Session service — creates sessions, serves session history and live status to the dashboard.
  • Deploy service — promotes an agent version to an environment, manages the stable endpoint mapping.
  • Tool registry — the catalog of available tools and their input schemas; supplies the worker with the set of callable tools for an agent version.
  • Skills registry — stores skill definitions and versions, resolves which skill versions an agent version loads.
  • Memory service — recall on the read path: retrieves the most relevant memories for a scope and serves them to the worker at session start.
  • Auth / RBAC — team membership and role scoping (who can edit vs. only view).
  • Secrets / vault — encrypted storage and retrieval of provider API keys.
  • Observability — ingests traces, metrics, and per-session cost data for the monitoring views.

06The execution layer

Agent runs — the loop of calling a model, executing a tool, calling the model again — happen in a separate worker pool fed by a queue, deliberately apart from the API servers.

This separation matters because agent runs are slow and bursty: a single run can take minutes and involve many model calls, while API requests need to return in milliseconds. Running both in the same process means one heavy agent run starves the dashboard. Splitting them lets the worker pool scale against queue depth, independent of API traffic.

The queue also provides durability for free. If a worker crashes mid-run, the task returns to the queue and another worker picks it up. For a multi-provider system this matters doubly, since provider rate limits and outages are routine — the queue absorbs them instead of surfacing them as user-facing failures. The same worker infrastructure runs the scheduled dreaming job (§09), so memory consolidation reuses the queue's retries and concurrency limits rather than introducing a second scheduler.

07Tools & processing

The agent worker decides to call a tool; the processing layer is where that call actually runs. Keep it separate from the provider adapter (§08): the adapter talks to models, the processing layer talks to tools.

  • Tool registry — the catalog of tools and their input schemas. The worker reads it to tell the model what's callable, and validates every tool call against the schema before running it.
  • MCP gateway — routes MCP tool calls to the right server and manages the connection and auth to each. One agent may use several MCP servers; the gateway is the single egress point, so connection handling and credentials aren't scattered across the worker.
  • Tool runtime / sandbox — where handlers execute. Code-running tools run in a sandbox (isolated container or microVM) with no ambient credentials, per-call timeouts, and concurrency caps.

Why keep this separate from the provider adapter? Symmetry. The provider adapter is one interface over many models; the processing layer is one interface over many tools. Conflating them tangles model-specific code with tool-specific code — splitting them means adding a tool never touches provider code, and adding a provider never touches tool code.

Tools are the blast radius. A model call only produces text; a tool call changes state — it writes a row, sends a request, or runs code. Treat every handler as untrusted by default: sandbox code execution, scope credentials to the specific tool (never hand the agent raw provider or database keys), set timeouts, and log every call as a tool_call/tool_result event so the session trace shows exactly what the agent did, not just what it said.

08The provider adapter

Everything above this layer speaks one internal interface. Only the adapter knows that Anthropic streams differently from OpenAI, that tool-call formats differ between providers, or that token counting works differently per model family.

interface LLMAdapter {
  stream(messages, config) -> AsyncIterator<Chunk>
  countTokens(messages) -> number
  toolFormat(tools) -> ProviderTools
  embed(texts) -> Vector[]          // memory recall reuses this
}

Concentrating provider-specific knowledge in one place means adding a new provider is a contained change to one layer, not a refactor spread across the codebase. The same adapter also produces the embeddings the memory subsystem recalls against, so there's one place that talks to providers — for completions and for embeddings alike.

Build vs. buy. Writing every adapter by hand gives maximum control but costs real time. A normalizing gateway — LiteLLM (proxy giving every provider an OpenAI-compatible API) or the Vercel AI SDK (TypeScript, unified streaming and tool calling) — gets the same abstraction with fallbacks, retries, and spend tracking built in. For a team-internal tool, starting on one of these saves weeks; hand-written adapters can replace it later if you outgrow it.

09Memory & dreaming

The memory subsystem has two paths that are deliberately asymmetric: recall is synchronous and on the hot path; formation is offline and scheduled.

Recall — the read path

At session start, and optionally mid-session, the memory service retrieves the most relevant memories for the active scope (agent, user, or team) and injects them into the agent's context. Relevance is vector similarity over the memory store, weighted by recency and salience. This is a control-plane read — fast and synchronous, sitting in the hot path of starting a session — so an agent begins already knowing what it learned before.

Formation — dreaming, the write path

Memory is not written synchronously during a session. Candidate observations accumulate, and a scheduled dreaming job — a repeatable worker task, off the hot path — processes them per scope: distil episodic summaries, extract semantic facts, dedupe against existing memories by similarity, prune stale or superseded ones, and recompute salience. Running it offline keeps the session loop fast, lets consolidation use a cheaper/slower model, and lets it look across many sessions at once — the cross-session view a synchronous per-session write can never have.

Storage

Memories live in Postgres with an embedding column (pgvector) for similarity recall. Promote to a dedicated vector store only when recall latency or index size demands it — the same "keep the boundary, migrate later" move as session_events → ClickHouse in the data layer.

Why dreaming is scheduled, not triggered per session. Consolidating ten sessions at once is both cheaper and better than ten separate consolidations: it can see that the same fact showed up five times (raise its salience), or that today's session contradicts last week's (supersede the old memory). A per-session write can't see the forest. Schedule it on idle — nightly, or after N new sessions — bound each run, and record it as a dream_run so every change to memory is auditable.

10The data layer

The data layer splits by access pattern, which is the detail most teams get wrong until volume forces the issue.

Entity data — agents, sessions, deployments, tools, skills — is low-volume, relational, and needs transactions: that's Postgres. Session traces — every message, token count, latency figure, tool call — are high-volume, append-only, and queried analytically ("cost per agent this week"). Forcing both into the same table works until trace volume swamps the database and dashboard queries slow down.

The fix is to keep traces in their own table from day one (session_events, see schema below) so the write pattern is already isolated, then migrate that table to a columnar store like ClickHouse — or partition it within Postgres — once volume actually demands it. You don't need ClickHouse on day one; you need the table boundary to already exist so the migration is a data move, not a redesign. Memory embeddings get the same treatment: an embedding column on memories in Postgres now, a dedicated vector store later, with the boundary already drawn.

Why agents and agent_versions are split

agents is an identity row that never changes. All real configuration lives in agent_versions, where every edit inserts a new row. A deployment's agent_version_id always points at something immutable — promoting a version is changing one foreign key, and rolling back is pointing it at the previous one.

Why tools and skills bind through join tables, not JSON columns

Tools and skills attach to an agent_version through agent_version_tools and agent_version_skills rather than a JSON list on the version. The join tables make the reverse query — "which agents use this tool?" or "which agents load this skill version?" — a simple index lookup, which is exactly the question you ask before changing or deprecating a shared tool. They also let a skill carry its own version lineage (skills → skill_versions) on the same append-only pattern as agents, so a session records the precise skill version it ran with.

Why sessions carries both agent_version_id and deployment_id

A session can originate from two paths: someone testing an agent directly in the builder (no deployment), or a deployed endpoint being invoked (has a deployment). Making deployment_id nullable lets one table serve both paths, so the session-monitoring view never needs a union query across two tables.

Why session_events is its own table, not a JSON column on sessions

It's tempting to add a messages jsonb column on sessions and append to it. Avoid this. Postgres rewrites the entire JSON blob on every append, which slows down as conversations grow, and a blob can't be queried at the event level — exactly the query you'll want once there's real usage ("show every failed tool call this week"). A separate table with an event_type column (message, tool_call, tool_result, error) keeps writes small and queryable, and is also the table to move into ClickHouse first when volume grows.

Why memories carries a scope and an embedding, and dream_runs exists

memories.scope_type + scope_id let the same table serve per-agent, per-user, and per-team memory without three near-identical tables — recall filters by scope, then ranks by vector similarity over embedding and by salience/last_used_at. dream_runs is the audit trail for formation: because dreaming mutates memory (writing, merging, pruning) outside any session, you want a record of every pass — what it processed and what it changed — so a memory that suddenly disappears or shifts is traceable to the run that did it, not a mystery.

Why provider_credentials stores encrypted_key, never a raw key

Provider API keys should never sit in plaintext, even in a database only the team can reach. Use envelope encryption: a master key held in a KMS (AWS KMS, GCP KMS, or HashiCorp Vault) encrypts each provider key before it's written, and the app decrypts on read via a KMS call rather than reading a stored key directly. A database leak alone then doesn't leak provider credentials.

Add later, not now: an index on (deployment_id, started_at) on sessions as soon as the monitoring view filters by deployment and time range — which will be almost immediately; a team_members join table with roles once more than one permission level is needed; a dedicated vector store for memories only once pgvector recall latency or index size becomes the bottleneck; per-tool usage counters once you need to rank tools by adoption rather than just list them.

12Scaling notes

In rough order of when each becomes a real problem:

  1. Trace volume overwhelming the primary database — mitigated by keeping session_events as its own table from the start, migrating to ClickHouse or partitioned Postgres when volume demands it.
  2. Long-running agent runs holding worker capacity — mitigated by scaling workers against queue depth and setting per-run timeouts.
  3. Provider rate limits cascading into failures — mitigated by the queue plus per-provider concurrency caps inside the adapter layer.
  4. Memory recall slowing the session hot path — mitigated by an embedding index in pgvector, scope-filtering before ranking, and graduating to a dedicated vector store only when the index outgrows Postgres.
  5. Dreaming cost growing with session volume — mitigated by running it on idle against a bounded batch, using a cheaper model for consolidation, and processing per scope rather than over the whole corpus each pass.
  6. Cost visibility lagging reality — mitigated by metering tokens at the adapter and streaming usage to the observability store in real time rather than in a nightly batch.

14Monorepo structure

A monorepo fits because the dashboard, API, and worker share the same TypeScript types (Agent, Session, Deployment, Tool, Skill, Memory), and a schema change should update all three without juggling package versions across separate repos.

Tooling: Turborepo + pnpm workspaces. Turborepo caches builds and runs tasks in parallel across packages without the configuration overhead of Nx — appropriate weight for a repo that won't have 15+ packages. Skip Lerna; it's legacy.

agent-platform/
├── apps/
│   ├── dashboard/      # Next.js — agent builder, session monitor, deploy UI
│   ├── api/            # Fastify/Hono — control plane (agents, sessions, tools, skills, memory recall)
│   └── worker/         # BullMQ — agent execution loop + scheduled dreaming job
├── packages/
│   ├── db/             # Drizzle schema + migrations + typed client
│   ├── adapter/        # Provider adapter — unified LLM interface + embeddings
│   ├── tools/          # Tool registry + runtime/sandbox + MCP gateway
│   ├── skills/         # Skill definitions, versioning, loading
│   ├── memory/         # Recall + dreaming/consolidation logic
│   ├── types/          # Shared TS types: Agent, Session, Tool, Skill, Memory, etc.
│   └── ui/             # Shared shadcn/ui components
├── turbo.json
├── pnpm-workspace.yaml
└── package.json

packages/db is the package every app touches — the dashboard reads sessions for monitoring, the API writes agents and deployments, the worker writes session_events and memories. Defining the schema once means a column rename surfaces as a type error everywhere it's used, instead of a runtime bug discovered later.

packages/adapter, packages/tools, and packages/memory stay separate from apps/worker so each can be unit-tested in isolation — mocking Anthropic vs. OpenAI responses, a tool handler, or a consolidation pass without spinning up the queue — and so the API can call them directly too: a "test this agent" button that shouldn't need the full async worker path, or a memory-recall call on the session read path.

15Build sequencing

Build the provider adapter and the Agent/Session models first, and get one agent running end to end with live session traces visible. Tools come in this same first push — an agent with no tools is just a chat box, so the tool registry and the processing/sandbox layer are part of "one agent running end to end," not a later add-on.

Memory and dreaming come next, once the core loop works and there are real sessions to consolidate — you can't dream without session history to dream over. Recall is the half to ship first (it's read-only and immediately useful); the scheduled dreaming job follows once there's enough history that consolidation has something to chew on.

Skills layer in once more than one agent needs the same capability — building the skills subsystem before that is premature abstraction. The builder UI and deployment come last.

Trying to build all of this fully in parallel tends to stall progress on all of it. The data model has to account for every capability from the start — hence pinned agent versions, join tables for tools and skills, and the memory tables in the schema above — but the build order doesn't have to match the data model's symmetry.
